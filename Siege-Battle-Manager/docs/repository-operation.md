# 저장소 운영 기준

이 문서는 SW 점령전 프로젝트의 공개 포트폴리오 저장소와 비공개 실서비스 저장소를 어떻게 나누어 운영할지 정리한다.

## 저장소 구분

### 공개 저장소

공개 저장소는 포트폴리오와 코드 리뷰 목적의 안정 버전으로 유지한다.

- README, 트러블슈팅, 배포 구조 설명, 주요 기능 설명을 관리한다.
- 민감값이 없는 소스 코드와 예시 설정만 포함한다.
- 운영 DB 접속 정보, JWT secret, SMTP 비밀번호, 관리자 IP, 운영 로그는 포함하지 않는다.
- 대규모 원본 데이터, 점령전 로그 샘플, 실서비스 전용 스크립트는 포함하지 않는다.

### 비공개 저장소

비공개 저장소는 실제 배포와 운영 기준으로 관리한다.

- Render, Cloudflare Pages 등 실제 배포 연결은 비공개 저장소를 기준으로 둔다.
- 운영 환경변수는 플랫폼 환경변수로 관리하고 Git에 커밋하지 않는다.
- 점령전 로그 통계, 원본 로그 파서, 운영 전용 자동화는 비공개 저장소에서 개발한다.
- 공개 저장소에 올리기 애매한 운영 문서와 절차를 관리한다.

## 작업 흐름

일반 기능 개발은 로컬에서 충분히 테스트한 뒤 비공개 저장소에 먼저 반영한다.

1. 로컬에서 백엔드와 프론트엔드를 실행해 기능을 확인한다.
2. 테스트와 빌드를 통과시킨다.
3. 비공개 저장소에 커밋하고 푸쉬한다.
4. Render와 Cloudflare Pages 배포 결과를 확인한다.
5. 포트폴리오에 공개해도 되는 안정 버전만 공개 저장소에 선별 반영한다.

## 공개 저장소 반영 기준

다음 조건을 만족하는 변경만 공개 저장소에 반영한다.

- 운영 비밀값이 포함되지 않는다.
- 실제 사용자 데이터나 운영 로그가 포함되지 않는다.
- 기능 설명이나 포트폴리오 가치가 있다.
- 배포 전용 설정이 민감 정보를 노출하지 않는다.
- 공개해도 악용 가능성이 낮은 코드와 문서다.

## 배포 연결 기준

- 백엔드 Render 서비스는 비공개 저장소의 배포 브랜치를 바라본다.
- 프론트 Cloudflare Pages도 비공개 저장소의 배포 브랜치를 바라본다.
- 공개 저장소 푸쉬는 포트폴리오 문서 갱신 용도로만 사용한다.
- 운영 환경변수는 Render, Cloudflare, Neon 대시보드에서 관리한다.

## 커밋과 푸쉬 기준

- 기능 단위가 작게 끝나면 Conventional Commit 형식의 한글 메시지로 커밋한다.
- 운영 배포가 필요한 변경은 비공개 저장소에 푸쉬한다.
- README나 포트폴리오 문서만 수정한 경우 공개 저장소에 푸쉬한다.
- 공개 저장소와 비공개 저장소를 동시에 푸쉬할 때는 민감 정보 포함 여부를 다시 확인한다.

## 다음 점검 항목

- 공개 저장소 README가 포트폴리오 관점에서 충분한지 확인한다.
- 비공개 저장소 배포 연결이 정상인지 주기적으로 확인한다.
- 새 운영 기능을 만들 때 공개 가능 여부를 먼저 분류한다.
- 정기적으로 `gitleaks` 또는 `trufflehog`로 저장소를 검사한다.
