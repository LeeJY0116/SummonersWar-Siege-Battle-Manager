# 🔐 JWT 인증 시 403 Forbidden 발생

## 문제 상황

로그인 이후:

```text id="g2n5e5"
GET /api/guilds/me
```
요청 시 403 Forbidden 발생

---

## 원인

프론트엔드에서 JWT 저장은 되었지만:

```text
Authorization: Bearer undefined
```

형태로 전송되고 있었음

즉 실제 토큰이 Header에 포함되지 않음

---

## 해결 과정
### 1. DevTools Network 확인

Request Header 분석

---

### 2. JWT 저장 로직 확인

로그인 성공 이후:

- localStorage 저장
- 토큰 반환값

확인 진행

---

### 3. apiFetch 공통 함수 수정
```javascript
headers.Authorization = `Bearer ${token}`;
```
형태로 공통 처리

---

## 결과
- 인증 정상 처리
- /api/guilds/me 조회 성공

---

# 🔐 Spring Security 인증은 성공하지만 404 발생
## 문제 상황

```text

```

응답 발생


즉:

- 로그인은 정상
- JWT 인증 정상
- 하지만 길드 가입 정보가 존재하지 않음

---

## 해결
- User 기준 GuildMember 조회 로직 점검
- 길드 생성 시 GuildMember 자동 생성 처리 추가

# ⚡ JPA N+1 문제 발생
## 문제 상황

방덱 목록 조회 시:

```text
DefenseDeck 조회
→ owner 추가 조회
→ monsters 추가 조회
```
반복 발생

---

## 원인

JPA Lazy Loading으로 인해 방덱 수 만큼 추가 SQL 발생

---

## 해결 방법

Fetch Join 적용
```java
@Query("""
select distinct d
from DefenseDeck d
join fetch d.owner
join fetch d.monsters
""")
```

---

## 결과
- 불필요한 SQL 제거
- 조회 성능 개선

---

# 🔄 Swarfarm 동기화와 로컬 관리 데이터 병합 문제
## 문제 상황

Swarfarm 동기화 후 기존에 숨김 처리한 몬스터가 다시 화면에 표시되는 문제가 발생

---

## 원인

외부 API는 전체 몬스터 데이터를 기준으로 내려주지만, 서비스에서는 실제 사용하는 몬스터만 노출해야 했음

또한 한글명/별칭은 외부 API가 아니라 로컬 관리 파일에서 관리해야 했음

---

## 해결

- Swarfarm 동기화는 기본 몬스터 정보와 이미지 URL만 갱신
- 신규 몬스터는 관리 파일에 `enabled: false`로 추가
- 화면 노출 여부는 `monster-localization.json`의 `enabled` 값을 우선 사용
- 한글명과 별칭은 관리 화면에서 직접 수정 가능하도록 구성

---

## 결과

외부 API 동기화와 로컬 운영 데이터가 충돌하지 않도록 분리

---

# 🖼️ 외부 이미지 파일 관리 문제

- 프론트는 저장된 URL을 사용해 이미지를 표시

---

## 결과

- UI 문구는 한글 직접 표기 사용
- 한글 포함 파일은 `apply_patch` 우선 사용
- 대량 변환은 UTF-8을 명시한 Node 스크립트만 사용
- 수정 후 깨짐 패턴 검색 실행

---

## 결과

# 🔐 가입 승인 전 로그인 가능 문제
## 문제 상황

길드장 또는 길드원 가입 신청이 승인되지 않은 상태에서도 로그인이 가능했다.

로그인 후 일부 기능은 권한이 없어 막혔지만, 전투 연구 작성처럼 사용자 계정만 있으면 접근 가능한 기능은 사용할 수 있었다.

---

## 원인

로그인 로직이 다음 두 가지만 확인하고 있었다.

- 사용자 존재 여부
- 비밀번호 일치 여부

길드 가입 승인 상태인 `PENDING`, `APPROVED`, `REJECTED`를 확인하지 않아 승인 전 사용자에게도 JWT가 발급되었다.

---

## 해결

`UserService.login()`에서 로그인 성공 전에 승인 상태를 확인하도록 변경했다.

- `ADMIN` 계정은 예외로 로그인 허용
- 일반 사용자는 `GuildMemberStatus.APPROVED`일 때만 로그인 허용
- 길드장 신청 대기 상태는 `관리자 승인 후 로그인할 수 있습니다.` 메시지 반환
- 길드원 신청 대기 상태는 `길드장 승인 후 로그인할 수 있습니다.` 메시지 반환
- 거절 상태는 `가입 요청이 거절되었습니다.` 메시지 반환

---

# 🏰 승인 전 길드가 가입 대상으로 노출되는 문제
## 문제 상황

길드장 가입 신청만 완료했을 뿐 아직 관리자가 승인하지 않았는데도 길드가 먼저 생성되었다.

그 결과 다른 사용자가 해당 길드 이름으로 길드원 가입 신청을 할 수 있었다.

---

## 원인

초기 구조에서는 길드장 회원가입 시점에 바로 다음 데이터가 생성되었다.

- `User`
- `Guild`
- `GuildMember(MASTER / PENDING)`

즉, 승인 전 길드도 실제 길드 데이터로 존재했다.

---

## 해결

1차로 승인된 마스터가 있는 길드만 가입 가능한 길드로 인정하도록 막았다.

이후 구조를 개선해 회원가입 시점에는 실제 `Guild`를 만들지 않고, `SignupRequest`만 저장하도록 변경했다.

관리자가 길드장 가입 신청을 승인할 때 실제 `User`, `Guild`, `GuildMember`를 생성하도록 변경했다.

---

# 🧾 가입 거절 후 같은 아이디 재신청 불가 문제
## 문제 상황

가입 신청을 거절했는데도 같은 아이디로 다시 신청하면 다음 오류가 발생했다.

```text
이미 사용 중인 아이디입니다.
```

---

## 원인

가입 신청 시점에 이미 `User`가 생성되어 있었다.

거절 처리는 `GuildMember.status`만 `REJECTED`로 변경했기 때문에, 실제 사용자 계정은 DB에 계속 남아 있었다.

그 결과 거절된 신청의 아이디, 이메일, 닉네임이 계속 점유되었다.

---

## 해결

`SignupRequest` 엔티티를 추가해 가입 신청과 실제 계정 생성을 분리했다.

- 가입 신청 시 `SignupRequest(PENDING)`만 생성
- 승인 시 실제 `User`, `Guild`, `GuildMember` 생성
- 거절 시 `SignupRequest`만 `REJECTED` 처리
- 거절된 신청은 실제 계정을 만들지 않으므로 같은 아이디로 재신청 가능

---

# 🔑 승인 대기 사용자의 로그인 메시지 부정확 문제
## 문제 상황

가입 신청만 존재하고 아직 승인되지 않은 사용자가 로그인을 시도하면 다음 메시지가 표시되었다.

```text
아이디 또는 비밀번호가 올바르지 않습니다.
```

하지만 실제로는 아이디와 비밀번호가 틀린 것이 아니라 승인 대기 상태였다.

---

## 원인

가입 신청 구조를 `SignupRequest`로 분리한 뒤, 승인 전 사용자는 아직 `users` 테이블에 존재하지 않는다.

기존 로그인 로직은 `users` 테이블만 조회했기 때문에, 승인 대기 신청자를 일반 로그인 실패로 처리했다.

---

## 해결

`UserService.login()`에서 실제 사용자 조회 실패 시 `SignupRequest(PENDING)`를 추가로 확인하도록 변경했다.

- 신청 아이디, 이메일, 닉네임 기준으로 대기 신청 조회
- 비밀번호가 맞으면 승인 대기 메시지 반환
- 비밀번호가 틀리면 기존 로그인 실패 메시지 유지

---

# 🧭 가입 승인 UI 위치 문제

가입 승인 UI가 길드 탭 상단 공통 영역에 표시되어, 인벤토리/방덱/전투 연구 화면에서도 계속 노출되었다.

기능의 성격은 회원 관리였지만, UI 구조는 모든 길드 하위 탭에 영향을 주는 위치였다.

---

## 해결

`회원 관리` 탭을 새로 만들고, 길드장과 부길드장에게만 표시되도록 분리했다.

회원 관리 탭 안에서 다음 기능을 함께 확인하도록 구성했다.

- 길드원 가입 승인/거절
- 현재 활성 길드원 목록

---

# 🌐 백엔드 API 변경 후 새 API가 없는 것처럼 보이는 문제
## 문제 상황

프론트에서 다음 API를 호출했을 때 오류가 발생했다.


Spring Boot 서버가 새 컨트롤러 매핑을 포함한 상태로 재시작되지 않으면, 해당 API가 존재하지 않는 것처럼 응답할 수 있다.

---

## 해결

새 API 추가 후 백엔드를 재시작하고, 빌드/테스트로 API 매핑을 확인했다.

- `./gradlew.bat test`
- 프론트 `npm run build`

# 🧪 정책 변경 후 테스트 실패 문제
## 문제 상황

가입 승인 정책을 변경한 뒤 기존 회원가입 테스트가 실패했다.

---

## 원인

기존 테스트는 회원가입 시점에 바로 다음 데이터가 생성된다는 전제를 가지고 있었다.

- `User`
- `Guild`
- `GuildMember`

하지만 새 정책에서는 회원가입 시점에 `SignupRequest`만 생성되고, 실제 데이터는 승인 시점에 생성된다.

---

## 해결

테스트를 새 정책 기준으로 수정했다.

- 회원가입 시 `SignupRequest(PENDING)`만 생성되는지 검증
- 관리자 승인 후 `User`, `Guild`, `GuildMember(MASTER / APPROVED)`가 생성되는지 검증
- 길드원 가입은 승인된 길드에만 신청 가능한지 검증
- 거절 후 같은 아이디로 다시 신청 가능한지 검증
- 승인 대기 로그인 메시지 검증

---

# 🏗️ User 생성자 접근 제한 문제
## 문제 상황

가입 승인 시 `GuildApprovalService`에서 `new User()`를 호출하자 컴파일 오류가 발생했다.

```text
User() has protected access in User
```

---

## 원인

`User` 엔티티의 기본 생성자는 JPA용으로 `protected` 접근 제한자를 사용하고 있었다.

`GuildApprovalService`는 다른 패키지에 있기 때문에 직접 생성자를 호출할 수 없었다.

---

## 해결

`User.create(...)` 정적 팩토리 메서드를 추가했다.

승인 서비스에서는 생성자를 직접 호출하지 않고, 정적 팩토리 메서드를 통해 `User`를 생성하도록 변경했다.

---

# 🔁 승인 API 대상 ID 의미 변경 문제
## 문제 상황

기존 승인 API는 `GuildMember.id`를 기준으로 승인/거절을 처리했다.

하지만 가입 신청을 `SignupRequest`로 분리한 뒤에는 승인 전 `GuildMember`가 존재하지 않는다.

---

## 원인

승인 전 실제 길드원을 만들지 않는 구조로 변경되면서, 승인 대상의 기준 ID가 `GuildMember.id`에서 `SignupRequest.id`로 바뀌었다.

---

## 해결

프론트 호환성을 위해 응답 필드명은 기존 `memberId`를 유지했다.

다만 내부적으로는 해당 값을 `SignupRequest.id`로 해석해 승인/거절을 처리하도록 변경했다.

---

# 🚫 재가입 불가 목록이 표시되지 않는 문제
## 문제 상황

추방 후 재가입 불가 목록에 추가되는 로직은 실행됐지만, 프론트 회원 관리 탭에서는 목록이 비어 보였다.

---

## 원인

다른 API는 공통 응답 포맷인 `ApiResponse.data`를 사용했지만, 재가입 불가 목록 API는 배열을 직접 반환했다.

프론트는 `response.data.data` 기준으로 읽고 있어 실제 배열을 찾지 못했다.

---

## 해결

프론트에서 다음 두 응답을 모두 처리하도록 보정했다.

- `ApiResponse.data`
- 배열 직접 응답

---

# ⚠️ 재가입 차단 상태에서 가입 요청 시 500 발생 문제
## 문제 상황

추방된 계정이 같은 길드에 다시 가입 요청을 하면 서버 내부 오류 메시지가 표시되었다.

---

## 원인

비즈니스 정책 위반을 `IllegalStateException`으로 던졌지만, 전역 예외 처리에서 해당 예외를 400으로 변환하지 않았다.

그 결과 사용자가 복구 가능한 요청 오류가 500 서버 오류로 응답되었다.

---

## 해결

`GlobalExceptionHandler`에서 `IllegalStateException`을 400 응답으로 처리하도록 추가했다.

---

# 🧍 탈퇴 후 재가입 시 같은 사용자가 중복 표시되는 문제
## 문제 상황

관리자 길드 관리 화면에서 탈퇴 후 재가입한 사용자가 여러 줄로 표시되었다.

---

## 원인

길드 이력 보존을 위해 `GuildMember` row를 삭제하지 않고 남기도록 설계했다.

하지만 관리자 목록 조회에서는 과거 row와 현재 row를 모두 같은 목록에 표시하고 있었다.

---

## 해결

관리자 길드원 목록은 같은 실제 유저 기준으로 최신 row 하나만 표시하도록 정리했다.

과거 길드 이동 내역은 별도 이력 API와 펼침 UI에서 확인하도록 분리했다.

---

# 🧭 현재 멤버십 조회 충돌 문제
## 문제 상황

사용자가 탈퇴 후 다른 길드에 재가입할 수 있게 되면서, `findByUser` 방식의 조회가 현재 소속을 정확히 판단하지 못할 수 있었다.

---

## 원인

한 유저가 여러 `GuildMember` row를 가질 수 있게 되었지만, 기존 코드는 유저당 길드원 row가 하나라는 전제를 가지고 있었다.

---

## 해결

현재 소속을 판단할 때는 `APPROVED` 상태의 최신 길드원 row를 기준으로 조회하도록 변경했다.

이전 row는 이력으로만 사용하고, 권한/인벤토리/방덱 접근은 현재 승인된 멤버십 기준으로 제한했다.

---

# 📝 닉네임 변경 이력 추적 문제
## 문제 상황

게임 내 닉네임은 변경될 수 있지만, 사이트에서 바로 수정하게 두면 이전 닉네임을 누가 사용했는지 추적하기 어렵다.

---

## 원인

닉네임은 길드 운영에서 식별자처럼 사용되기 때문에 무제한 직접 변경을 허용하면 악용 가능성이 있었다.

---

## 해결

닉네임 변경을 요청/승인 구조로 분리했다.

- 사용자는 닉네임 변경 요청 생성
- 관리자는 승인/거절
- 승인 시 `User.nickname`과 현재 `GuildMember.displayName` 갱신
- 변경 이력은 `UserNicknameHistory`에 저장
- 관리자 화면에서 닉네임 이력을 펼쳐 확인 가능

---

# 🔎 전투 연구 필터가 현재 페이지에만 적용될 수 있는 문제
## 문제 상황

전투 연구 게시글이 많아지면 10개 단위 페이지네이션이 필요하다.
이때 프론트에서만 검색/필터를 처리하면 현재 내려받은 페이지 안에서만 결과가 검색될 수 있다.

---

## 원인

프론트 페이징은 이미 받아온 데이터를 화면에서만 나누는 방식이고,
백엔드 페이징은 조회 조건을 서버에 전달해 전체 데이터 중 필요한 페이지를 가져오는 방식이다.

전투 연구는 빠르게 10개를 초과할 수 있으므로 프론트 전용 필터링은 실제 운영 검색과 맞지 않는다.

---

## 해결

전투 연구 목록 조회 시 다음 조건을 백엔드 요청 파라미터로 전달하도록 변경했다.

- 페이지 번호
- 페이지 크기
- 리더 효과
- 포함 몬스터
- 4성 방덱 여부

백엔드에서 전체 게시글 기준으로 필터링한 뒤 페이지네이션해, 아직 조회하지 않은 페이지의 게시글도 검색 결과에 포함되도록 했다.

---

# 🧩 방덱 소유 길드원 변경 시 선택 몬스터가 섞이는 문제
## 문제 상황

방덱 생성 화면에서 A 길드원의 몬스터를 선택한 뒤 소유 길드원을 B로 바꿔도 기존 선택 몬스터가 남아 있었다.

---

## 원인

방덱 생성 폼의 소유자 상태와 선택 몬스터 슬롯 상태가 독립적으로 관리되고 있었다.
소유자 변경 시 기존 선택 슬롯을 초기화하지 않아 서로 다른 길드원의 몬스터가 한 방덱에 섞일 수 있었다.

---

## 해결

방덱 소유 길드원이 변경되면 선택 몬스터 슬롯, 현재 선택 슬롯, 검색어를 초기화하도록 수정했다.

---

# 🛡️ 길드 방덱 생성 가능 인원에 탈퇴자가 포함되는 문제
## 문제 상황

길드 방덱 생성 가능 길드원 수에 이미 탈퇴한 길드원이 계속 포함되었다.

---

## 원인

생성 가능 인원을 계산할 때 현재 승인된 길드원만 필터링하지 않고, 과거 이력 row까지 함께 계산할 수 있었다.

---

## 해결

`GuildMemberStatus.APPROVED` 상태인 현재 길드원만 생성 가능 인원 계산 대상으로 사용하도록 수정했다.

---

# 🌏 Render 백엔드 리전과 Neon DB 리전 불일치로 인한 지연
## 문제 상황

운영 배포 후 로그인, 인벤토리, 방덱, 전투 연구 API가 전반적으로 1초 이상 지연되었다.

---

## 원인

Neon PostgreSQL은 Singapore 리전에 있었지만, Render 백엔드는 Oregon 리전에 생성되어 있었다.

사용자 요청은 프론트에서 백엔드로 빠르게 도착하더라도, 백엔드가 DB를 조회할 때마다 미국 서부와 싱가포르 사이를 왕복하면서 지연이 발생했다.

---

## 해결

Render 백엔드를 Singapore 리전에 새로 생성하고 Cloudflare Pages의 `VITE_API_BASE_URL`을 새 백엔드로 변경했다.

변경 후 주요 API 응답 시간이 약 1000~1500ms대에서 100~300ms대로 줄어들었다.

운영 기준:

- 백엔드와 DB 리전은 같은 지역으로 맞춘다.
- Render는 기존 서비스 리전을 직접 변경할 수 없으므로 새 서비스를 만들고 환경변수를 이전한다.
- 새 백엔드 검증 후 기존 리전 서비스는 삭제해 중복 과금을 막는다.

---

# 🧵 몬스터 동기화와 도감 정보 적용이 장시간 요청으로 실패하는 문제
## 문제 상황

운영 환경에서 `몬스터 동기화` 또는 `도감 정보 적용`을 실행하면 오래 걸리거나, PostgreSQL I/O 오류가 발생했다.

---

## 원인

외부 API 호출과 대량 DB 갱신을 하나의 HTTP 요청/트랜잭션처럼 처리하면 Render, Neon, 네트워크 상태에 따라 요청이 길어지고 커넥션이 끊길 수 있었다.

또한 사용자가 버튼을 여러 번 누르면 같은 작업이 중복 실행될 가능성도 있었다.

---

## 해결

관리자 작업을 백그라운드 작업으로 분리하고 작업 상태를 조회하는 구조로 변경했다.

- 작업 시작 API는 즉시 응답
- 실제 동기화/적용은 백그라운드에서 진행
- 진행 상태, 처리 수, 실패 사유를 별도 상태 API로 확인
- 같은 작업의 중복 실행 방지
- `몬스터 동기화`와 `도감 정보 적용` 동시 실행 방지
- 서버 재시작 등으로 작업이 끊기면 실패 상태로 표시 후 다시 실행 가능

---

# 🧩 길드 방덱 중복 판정에서 리더가 무시되는 문제
## 문제 상황

길드 방덱 탭에서 리더가 다른데 나머지 몬스터 종류가 같으면 동일 방덱처럼 취급되는 문제가 있었다.

---

## 원인

중복 판정에서 방덱 전체 구성을 비교할 때 리더 슬롯의 의미가 충분히 반영되지 않았다.

점령전 방덱은 0번 몬스터가 리더이며, 같은 3마리라도 리더가 다르면 리더 효과와 운용 목적이 달라진다.

---

## 해결

중복 판정 기준을 `리더 + 나머지 두 몬스터의 정렬된 조합`으로 정리했다.

사용자에게 보여주는 순서는 사용자가 선택한 순서를 유지하고, 내부 비교용 키에서만 리더를 고정한 뒤 나머지 두 몬스터를 정렬한다.

---

# 🔐 운영 보안 점검과 저장소 분리
## 문제 상황

포트폴리오 공개 저장소와 실제 운영 저장소를 같은 기준으로 관리하면, 운영 환경변수나 DB 접속 정보가 GitHub에 올라갈 위험이 있었다.

---

## 원인

초기 개발 단계에서는 H2와 로컬 환경 중심으로 작업했지만, 운영 배포를 시작하면서 JWT secret, DB URL, Render/Cloudflare/Neon 환경변수처럼 공개되면 안 되는 값들이 생겼다.

---

## 해결

공개 포트폴리오 저장소와 비공개 실서비스 저장소를 분리했다.

- 실제 운영 배포는 비공개 저장소 기준으로 연결
- `.env`, 운영 DB 접속 정보, JWT secret은 Git 추적 대상에서 제외
- `render.yaml`의 민감 환경변수는 `sync: false`로 관리
- `gitleaks`, `trufflehog`로 히스토리 스캔
- 운영 smoke test와 인증 포함 smoke test를 문서화
- admin API, CORS, rate limit, H2 console 비활성화 여부를 점검

# 📌 배운 점
## 1. 인증 문제와 데이터 문제는 다르다

- 몬스터 수량 감소

- 권한 검증

등 상태 기반 로직 설계가 중요하다는 것을 체감
